# Review Artifacts and Phase-Check Hardening

**Version:** 1.1
**Created:** March 12, 2026
**Revised:** March 13, 2026
**Status:** Ready for implementation

## Overview

The harness-native implementation workflow is working well, but a few mechanical
rough edges still create operator friction and reduce auditability:

- reviewer prompts require an explicit `review_path`, so operators currently
  have to thread it through manually or pass awkward placeholders
- quality-fix author retries reuse the implementation-review template and pass
  `review_path=""`, which leaks the wrong mental model into the prompt
- native author agents often return `status: "done"` instead of the canonical
  `result: "complete"`, forcing manual normalization before
  `5x protocol validate`
- the phase-execution skill says `5x plan phases` is authoritative for checklist
  completion, but the workflow does not actually verify that invariant before
  advancing
- native-first skills still mention `5x run watch`, even though that guidance is
  tied to `5x invoke` log streaming and is misleading in the native path

This change hardens the workflow around those points without changing the core
author -> quality -> review -> fix loop. Review artifacts become persistent by
default, the quality-fix path gets its own template, author protocol inputs are
normalized at the validation boundary, and the orchestrator explicitly stops and
escalates if a phase's checklist was not updated.

## Design Decisions

**Default review paths are generated by the shared template-variable resolver.**
`5x template render` and `5x invoke` already share internal variable injection in
`src/commands/template-vars.ts`. That is the right place to auto-populate
`review_path` when a template declares it and the caller did not pass one.
Plan-review templates use `config.paths.planReviews ?? config.paths.reviews`;
implementation-review templates use `config.paths.runReviews ??
config.paths.reviews`. Explicit `--var review_path=...` still wins.

**Review artifacts should have stable, deterministic filenames.** Plan reviews
should use the full plan basename to avoid collisions across similarly named
plans. Run reviews should include both run id and phase number so each phase has
one persistent review document that can accumulate addenda. Recommended shapes:

- plan review: `<planReviews>/<full-plan-basename>-review.md` (basename provides
  stable identity; collisions across plans with same filename in different dirs
  resolved by using full relative path with separators replaced)
- implementation review: `<runReviews>/<run-id>-phase-<phase>-review.md`

These paths remain relative to the repository root unless configured as absolute
paths, matching existing config semantics. When the phase context is unavailable,
the resolver falls back to `<runReviews>/<run-id>-review.md` (one document per
run). For fresh reviews, operators can explicitly override with
`--var review_path=...`.

**Quality-fix retries need a dedicated author template.** Reusing
`author-process-impl-review` for quality failures creates a fake dependency on a
review document and forces `review_path=""` through the system. A dedicated
template keeps the author instructions accurate: fix failing tests/lint/type
checks, update plan checklist items only if needed, commit, and return a normal
`AuthorStatus`.

**Author protocol normalization belongs at the validation boundary.** The
workflow should continue to treat `result` as the canonical field name, but
`validateStructuredOutput()` should accept legacy native-agent payloads with
`status: "done" | "failed" | "needs_human"` and normalize them before running
the normal invariants. This removes manual operator work while keeping the
public protocol strict on output.

**Phase checklist verification should fail closed and escalate to the human.**
Before recording `phase:complete`, the orchestrator should run
`5x plan phases $PLAN_PATH` and verify the current phase reports `done: true`.
If verification fails, the workflow should stop, record `phase:checklist_mismatch`,
and escalate to the human rather than recording a misleading `phase:complete` or
silently continuing. This keeps the audit trail honest without introducing new
author-session-resume complexity.

**Remove `run watch` guidance from native-first skills.** `5x run watch` is a
useful transport-specific tool for `5x invoke` log streaming, but native
subagent workflows do not record equivalent logs through `5x protocol validate`.
Leaving the guidance in the skills suggests a monitoring path that does not map
cleanly to the native experience. The skills should stay focused on the native
orchestrator contract and mention `5x invoke` only as a fallback transport.

## Config Path Resolution Contract

All `paths.*` values in `5x.toml` are resolved to **absolute paths** by the CLI
after config loading, regardless of whether they were written as relative or
absolute in the config file:

- **Relative paths** in a config file are resolved against the config file's
  directory (e.g., `paths.plans = "docs/dev"` in
  `packages/foo/5x.toml` resolves to `<repo>/packages/foo/docs/dev`).
- **Zod default values** (when no explicit path is configured) are resolved
  against the workspace/project root.
- **Absolute paths** pass through unchanged.
- This applies to all config-loading entry points: `loadConfig()`,
  `resolveLayeredConfig()`, and any function that returns a `FiveXConfig`.

Callers always receive absolute `paths.*` values and should never need to resolve
paths themselves. This contract was introduced in
019-orchestrator-improvements Phase 1.

## Template Variable Defaults

Templates can declare default values for variables using the `variable_defaults`
key in YAML frontmatter. This allows templates to render without explicit
`--var` for variables that have sensible defaults (e.g., `user_notes: ""`).

### Syntax

```yaml
---
name: my-template
version: 1
variables: [required_var, optional_var]
variable_defaults:
  optional_var: ""
---
```

### Validation Rules

- `variable_defaults` must be a plain YAML mapping (not an array or scalar).
- Every key in `variable_defaults` must exist in the `variables` list.
- All values must be strings.
- Invalid keys or non-string values produce a parse-time error.

### Precedence

1. **Explicit `--var`** — always wins over defaults.
2. **`variable_defaults` value** — used when the variable is not provided
   explicitly.
3. **Missing-variable error** — raised for variables with no explicit value
   and no default.

Templates without `variable_defaults` behave exactly as before (all variables
are required). This feature was introduced in 019-orchestrator-improvements
Phase 2.

## Quality Gate Skip Configuration

The `skipQualityGates` config key controls behavior when no quality gates are
configured:

```toml
# 5x.toml
skipQualityGates = true   # Intentionally skip quality gates (silent)
```

### Behavior

| `skipQualityGates` | `qualityGates` | Result |
|--------------------|----------------|--------|
| `false` (default)  | empty `[]`     | Warning on stderr: "no quality gates configured…"; output: `{ passed: true, results: [] }` |
| `true`             | empty `[]`     | Silent; output: `{ passed: true, results: [], skipped: true }` |
| `false`            | non-empty      | Normal gate execution |
| `true`             | non-empty      | Normal gate execution (non-empty gates always run) |

- **Type:** boolean
- **Default:** `false`
- **`skipped: true`** in the output JSON allows downstream consumers (e.g.,
  the phase-execution skill) to distinguish intentional skips from actual
  gate results.

This feature was introduced in 019-orchestrator-improvements Phase 3.

## Not In Scope

- Adding new `postCreate` guardrails or changing worktree initialization policy
- Auto-reinvoking the author on checklist mismatch
- Adding author-session reuse to the phase-execution workflow
- Capturing native-subagent streaming logs equivalent to `5x invoke`
- Changing run database schema for review-path persistence
