# Review: Mixed-Mode Delegation for Native Harnesses

**Review type:** docs/development/plans/019-mixed-mode-delegation.md
**Scope:** Plan, shared skill renderer/templates, harness install path, plugin describe/uninstall behavior, and current harness tests.
**Reviewer:** Staff engineer
**Local verification:** Not run (static review only)

## Summary

The feature direction is sound and fits the existing shared-skill architecture, but the plan is not implementation-ready yet. Two blockers remain: the plan's cross-cutting template strategy is internally inconsistent with the proposed renderer semantics, and the agent-installation/reporting phase assumes config-aware `describe()` / uninstall behavior that the current plugin contract cannot support safely.

**Readiness:** Not ready — one implementation gap is mechanical but one install/reconciliation issue needs a design decision.

## Strengths

- Good architectural fit: per-role delegation extends the existing shared renderer and harness install flow instead of inventing a parallel orchestration stack.
- Correct default posture: native/native remains the default, so mixed mode is opt-in and backward compatibility is an explicit goal.
- The phase split is mostly sensible: config/context first, renderer second, templates after, then harness integration.

## Production Readiness Blockers

### P0.1 — Mixed-mode cross-cutting sections are not expressible with the proposed predicate model

**Classification:** auto_fix

**Risk:** The design says legacy `native` remains true only when both roles are native and `invoke` only when both are invoke, but multiple Phase 3 tasks then use those flags as if they meant “at least one native role” or “at least one invoke role.” In mixed mode that content would disappear or the implementation would have to invent behavior not defined in Phase 1/2.

**Requirement:** Tighten the plan so the predicate semantics and the template refactor strategy match. Either:
- add explicit mixed/cross-cutting predicates to `SkillRenderContext` and Phase 2, or
- rewrite the Phase 3 template strategy to use only predicates that are actually available under the proposed renderer semantics.

**Evidence:**
- Predicate semantics: `docs/development/plans/019-mixed-mode-delegation.md:73-80`, `124-136`, `160-163`
- Conflicting mixed-mode usage of `native`/`invoke`: `docs/development/plans/019-mixed-mode-delegation.md:193-218`, `235-253`, `267-270`
- Current renderer only supports exact boolean selectors, no OR/derived mixed-mode expression: `src/skills/renderer.ts:1-73`

### P0.2 — Config-driven `describe()` / uninstall accuracy is incompatible with the current harness contract

**Classification:** human_required

**Risk:** Phase 4 says plugin `describe()` should reflect which agents are installed “based on delegation mode” so `harness show` and uninstall stay accurate. But `describe(scope?)` currently has no config input, and `harnessList` / uninstall call it without loading config. Making uninstall depend on current config is also unsafe: if a user installed native reviewer assets, then later flips `reviewer.delegationMode = "invoke"`, config-aware uninstall could stop removing the already-installed reviewer file and leave stale managed assets behind.

**Requirement:** Decide the ownership model before implementation. At minimum, the plan must specify one of these:
- keep `describe()` as the static bundled asset inventory and handle mixed-mode installed-state reporting with filesystem inspection / install manifests, while uninstall removes all managed assets regardless of current config, or
- redesign the plugin/list/uninstall contract so config at install time is persisted and later used for accurate reconciliation.

**Evidence:**
- Proposed config-aware `describe()` change: `docs/development/plans/019-mixed-mode-delegation.md:327-329`
- Current plugin contract: `src/harnesses/types.ts:111-131`
- Current list path uses `plugin.describe(scope)` with no config: `src/commands/harness.handler.ts:204-246`
- Current uninstall also relies on `describe()` names to decide what to remove: `src/harnesses/opencode/plugin.ts:90-95`, `src/harnesses/cursor/plugin.ts:105-119`

## High Priority (P1)

### P1.1 — The template/test plan should explicitly cover harness-list and stale-asset behavior after config changes

**Classification:** auto_fix

Once mixed mode suppresses some agent files, the risk is no longer just render correctness; it is lifecycle correctness across reinstall, list, and uninstall after `5x.toml` changes. The current plan adds renderer/install tests, but not the behavioral cases that matter most operationally: install native/native, switch one role to invoke, reinstall, verify the skipped role's agent is removed or intentionally preserved, and verify `harness list` reports the resulting filesystem accurately.

**Recommendation:** Add explicit tests around config transitions and stale managed assets in the harness command/plugin test suites.

## Medium Priority (P2)

- Phase 3 calls the all-four-context rendering checks “integration-level tests,” but the existing skill renderer/loader coverage lives in unit tests (`test/unit/skills/renderer.test.ts`, `test/unit/skills/loader.test.ts`). Unless subprocess behavior is being exercised, keep these as unit tests and reserve integration tests for `5x harness install` end-to-end filesystem assertions. **Classification:** auto_fix
